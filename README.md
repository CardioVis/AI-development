# CardioVis AI-development (FixMatch-style Semi-supervised Segmentation)

This repository contains a **FixMatch-style** semi-supervised training pipeline for **multi-class medical image segmentation** on CardioVis frame data, where only a small subset of frames are annotated and the rest are used as **unlabeled** examples.

> Status note (important): this repo snapshot includes the training and dataset code, but **references model code that is not present** (see [Known gaps / intended components](#known-gaps--intended-components)). The instructions below describe the **intended** setup rigorously, and also call out what must be provided for the code to run end-to-end.

## Technical objectives

- **Task**: multi-class semantic segmentation of CardioVis frames.
- **Classes** (as used by config and preprocessing):
  - `0`: background
  - `1`: Epicardial adipose tissue
  - `2`: Pericardium
  - `3`: Phrenic nerve
- **Learning setting**: semi-supervised (labeled + unlabeled).
  - Labeled set: frames with merged 4-class masks.
  - Unlabeled set: frames without masks.
  - Pseudo-labels: derived from **weakly augmented** unlabeled images; enforced on **strongly augmented** versions using a confidence threshold.
- **Evaluation protocol**: 5-fold workflow (fold lists are generated by preprocessing; training script iterates folds).
- **Metrics**: Dice, IoU, Hausdorff distance (MONAI).

## Repository structure

- `fixmatch.py`: main FixMatch training / validation / testing loop (k-fold).
- `prepare_cardio_data.py`: converts raw CardioVis frames + per-class masks into `.npy` tensors and split lists.
- `Configs/cardio.yml`: main configuration for CardioVis paths and hyperparameters.
- `Datasets/`: dataset wrappers + strong/weak augmentation pipelines.
  - `Datasets/create_dataset.py`: reads fold lists + unlabeled list and loads `.npy` samples.
- `Losses/`, `Utils/`: supporting code.

## Data (private) and expected formats

The CardioVis dataset is treated as **private/internal**. This README documents **expected file formats and folder structure**, but does not include download links.

### Raw inputs expected by `prepare_cardio_data.py`

`prepare_cardio_data.py` is currently written with Windows-style default paths and expects:

- **Frames directory**: a folder containing frame images named like `frame_01044.jpg` (or `01044.jpg`), etc.
- **Label directory**: a folder containing per-class mask images named like:
  - `task-1-annotation-...-tag-Epicardial adipose tissue-0.png`
  - `task-1-annotation-...-tag-Pericardium-0.png`
  - `task-1-annotation-...-tag-Phrenic nerve-0.png`

The script maps `task-K` to `frame_(FRAME_OFFSET + K)` using:

- `FRAME_OFFSET = 1043` (so `task-1 -> frame_01044`)
- `FRAME_ID_DIGITS = 5` (zero-padded)

It merges the per-tag masks into a **single multi-class mask** with values 0–3.

### Processed dataset layout (what training expects)

After preprocessing, the code expects the following structure (default paths match `Configs/cardio.yml`):

```text
data_processed/
  cardio/
    train/
      labeled_data/
        images/
          01044.npy
          01045.npy
          ...
        labels/
          01044.npy
          01045.npy
          ...
        fold_label_1.txt
        fold_label_2.txt
        fold_label_3.txt
        fold_label_4.txt
        fold_label_5.txt
      unlabeled_data/
        images/
          01099.npy
          01100.npy
          ...
        unlabeled.txt
```

- Each `.npy` in `images/` is an **H×W×3** uint8-like array (saved from OpenCV image read).
- Each `.npy` in `labels/` is an **H×W** integer mask (0–3).
- `fold_label_k.txt` contains one filename per line (e.g., `01044.npy`), defining that fold’s labeled samples.
- `unlabeled.txt` contains one filename per line (e.g., `01100.npy`) for unlabeled frames.

## Preprocessing: build the processed dataset

1) Edit paths in `prepare_cardio_data.py` to point to your private data locations:

- `FRAMES_DIR`
- `LABEL_DIR`

2) Run preprocessing from the repo root:

```bash
python prepare_cardio_data.py
```

Outputs will be written under:

- `data_processed/cardio/train/labeled_data/`
- `data_processed/cardio/train/unlabeled_data/`

## Training (FixMatch-style)

### Configuration

The provided config is `Configs/cardio.yml`. Key fields:

- `data.num_classes`: `4`
- `data.train_folder`: `data_processed/cardio/train/labeled_data`
- `data.unlabeled_folder`: `data_processed/cardio/train/unlabeled_data`
- `data.save_folder`: `checkpoints/cardio`
- `semi.conf_thresh`: pseudo-label confidence threshold (default `0.95`)
- `train.{l_batchsize,u_batchsize,num_epochs,num_workers}`

### Run training

From the repo root:

```bash
python fixmatch.py --exp cardio_fixmatch --config_yml Configs/cardio.yml --gpu 0
```

What happens:

- The script loads the config YAML, seeds RNGs, and selects GPU if available.
- It loops through folds `1..5` (hard-coded in `fixmatch.py`) and writes outputs per fold to:

```text
checkpoints/cardio/<exp>/fold<k>/
  best.pth
  exp_config.yml
  log.txt
  test_results.txt
```

### Evaluation / testing

The script always runs a test phase after training each fold and writes metrics to `test_results.txt` (Dice / IoU / HD / loss).

To run “test-only”, set in your YAML:

```yaml
test:
  only_test: True
  test_model_dir: "/path/to/best.pth"
```

> Implementation note: `fixmatch.py` currently uses `best_model_dir` computed per fold and loads from that path inside `test()`. If you plan to use `test.test_model_dir`, you’ll likely want to adapt `fixmatch.py` accordingly.

## Reproducibility

- Seeding is performed via `Utils/utils.py::fix_all_seed(seed)` and also sets CUDA seeds.
- For exact reproducibility, consider controlling CUDA determinism flags and logging the exact environment (CUDA, driver, torch build).

## Known gaps / intended components

This repository snapshot is not fully self-contained:

- **Missing model package**: `fixmatch.py` imports `from Models.DeepLabV3Plus import *` and calls:
  - `deeplabv3plus_resnet101(num_classes=..., output_stride=8, pretrained_backbone=True)`

  You must provide a `Models/DeepLabV3Plus.py` (or equivalent package) that defines this function (and any required classes) for training to run.

- **Stale references in the docstring / defaults**:
  - `fixmatch.py` docstring references `multi_train_adapt.py` and `Configs/multi_train_local.yml`, which are not present here.
  - The actual usable config shipped is `Configs/cardio.yml`.

If you want this repo to be runnable “as-is”, the recommended cleanup is:

- Add the missing `Models/` code, and
- Remove/update references to non-existent scripts/configs in `fixmatch.py`.
